# ROS2 Integration

The core library (`ame_core`) is ROS-agnostic. The `ros2/` directory is a separate `ament_cmake` package (ROS2 Jazzy) that wraps the core with four lifecycle nodes.

## PCL Component Integration

Each ROS2 node owns a PCL-backed component (see `doc/architecture/08-pcl-component-system.md`). The ROS2 lifecycle states delegate directly to PCL lifecycle methods, ensuring 1:1 compatibility:

```cpp
// Pattern: ROS2 LifecycleNode wraps a pcl::Component subclass
class WorldModelNode : public rclcpp_lifecycle::LifecycleNode {
  ame::WorldModelComponent component_;  // extends pcl::Component

  CallbackReturn on_configure(const rclcpp_lifecycle::State&) override {
    component_.setParam("domain.pddl_file", get_parameter(...).as_string().c_str());
    if (component_.configure() != PCL_OK) return CallbackReturn::FAILURE;
    // Create ROS2 services that delegate to component_ methods
    srv_get_fact_ = create_service<GetFact>("~/get_fact", ...);
    return CallbackReturn::SUCCESS;
  }

  CallbackReturn on_activate(const rclcpp_lifecycle::State&) override {
    if (component_.activate() != PCL_OK) return CallbackReturn::FAILURE;
    // Create ROS2 publishers
    pub_world_state_ = create_publisher<WorldState>("/world_state", ...);
    return CallbackReturn::SUCCESS;
  }
};
```

This pattern keeps business logic in `pcl::Component` subclasses (portable, testable without ROS2) while ROS2 nodes handle middleware concerns (services, publishers, lifecycle coordination).

**Example:** `ros2/src/apps/combined_main.cpp` demonstrates all three nodes sharing one executor with direct `WorldModel*` access (zero-copy, no service IPC).

## Nodes

- **WorldModelNode** (`ros2/src/nodes/world_model_node.cpp`) — owns canonical `WorldModel`; exposes `~/get_fact`, `~/set_fact`, `~/query_state`, `~/load_domain` services; publishes `/world_state` (reliable, transient_local) on every fact change (debounced). WmAuditLog wired in `on_configure`. The `~/load_domain` service (`ame_ros2/srv/LoadDomain`) accepts PDDL domain+problem as strings and replaces the world model schema at runtime — used by the devenv for testing. In deployment, the union domain is loaded from file params at configure time.
- **PlannerNode** (`ros2/src/nodes/planner_node.cpp`) — `rclcpp_action::Server` at `~/plan` (e.g. `/planner_node/plan`). Snapshots world state → BRFS solve on dedicated `std::thread` → returns compiled BT XML. Exposes `~/load_domain` service for runtime domain loading from PDDL strings (bypasses file-path dependency). PlanAuditLog records each episode. Multiple PlannerNode instances can run concurrently with different node names and different domain models.
- **ExecutorNode** (`ros2/src/nodes/executor_node.cpp`) — owns BT factory + tree. Ticks via `rclcpp::Timer` at 50 Hz. AmeBTLogger publishes to `/executor/bt_events`. Supports direct `WorldModel*` (in-process) or service-backed BT nodes (distributed).
- **AgentDispatcherNode** (`ros2/src/nodes/agent_dispatcher_node.cpp`) — wraps `AgentDispatcher` PCL component for multi-agent coordination. Exposes `~/dispatch_goals` service (`ame_ros2/srv/DispatchGoals`). Dynamically creates per-agent publishers (`/{agent_id}/executor/bt_xml`) and status subscriptions. Used in multi-agent deployments for leader-delegation goal dispatch.
- **AmeLifecycleManager** (`ros2/src/lib/lifecycle_manager.cpp`) — configures then activates WorldModel → Planner(s) → Executor(s) (→ AgentDispatcher in multi-agent mode) in order.

## Deployment Modes

### In-process (single executor)

All nodes share one `SingleThreadedExecutor`. `setInProcessWorldModel(WorldModel*)` skips service calls entirely — no mutex needed since everything is single-threaded.

### Distributed (service-backed)

BT nodes use `RosCheckWorldPredicate` / `RosSetWorldPredicate` which call WorldModelNode services with 500ms timeout. Nodes can run on separate machines.

### Multi-agent

One `WorldModelNode`, one `PlannerNode`, one `AgentDispatcherNode`, and N `ExecutorNode` instances (one per agent). Each executor's topics are namespaced by `agent_id` (`/{agent_id}/executor/bt_xml`, `/{agent_id}/executor/status`). The dispatcher allocates goals and sends compiled BT XML to agent executors via `~/dispatch_goals` service.

### Multi-planner

One `WorldModelNode` (union domain), N `PlannerNode` instances (each with its own domain), and N `ExecutorNode` instances. Each PlannerNode has a unique node name (e.g. `planner_logistics`, `planner_surveillance`) and gets its own namespaced endpoints:

- `~/plan` → `/planner_logistics/plan` (action server)
- `~/load_domain` → `/planner_logistics/load_domain` (domain loading service)
- `~/bt_xml` → `/planner_logistics/bt_xml` (BT XML publisher, configurable via `bt_xml_topic` parameter)

The WorldModelNode loads the **union domain** (all predicates/objects from all planners). In deployment this is a file-based domain loaded at configure time. In devenv/testing, the union domain can be pushed at runtime via the WorldModelNode's `~/load_domain` service.

Each planner snapshots the shared world state via `QueryState`, then applies it to its own domain template — facts that don't match the planner's predicates are silently ignored. See `ros2/launch/ame_multi_planner.launch.py` for example configuration.

## Extension BT Nodes

`ExecutorNode` auto-registers three extension BT node types:

- **`InvokeService`** (Extension 4) — async PYRAMID SDK calls. Requires `IPyramidService*` via `setPyramidService()`.
- **`ExecutePhaseAction`** (Extension 6) — hierarchical sub-planning. Requires planner/compiler/registry on blackboard via `setPlanner()`, `setPlanCompiler()`, `setActionRegistry()`.
- **`DelegateToAgent`** (Multi-Agent) — leader-delegation. Same blackboard keys as `ExecutePhaseAction`.

In-process mode: `combined_main.cpp` wires all dependencies from `PlannerNode` accessors. Distributed mode: inject a `PlannerComponent*` via `setPlannerComponent()`.

## Design Constraints

- All nodes are `rclcpp_lifecycle::LifecycleNode`, coordinated by `AmeLifecycleManager`
- `ame_core` is completely unchanged — ROS2 is purely an adapter layer
- Perception nodes are independent writers calling `set_fact` on the WorldModelNode service

## Build

**Quick (use `build_ros2.bat`):**
```bat
build_ros2.bat
```

**Step by step (from a pixi shell at D:\Dev\ros2-windows):**
```bat
set PATH=D:\Dev\ros2-windows\.pixi\envs\default\Library\bin;%PATH%

:: 1. Install ame_core
cmake --preset default -DCMAKE_INSTALL_PREFIX=build/install
cmake --build build --config Release -j%NUMBER_OF_PROCESSORS%
cmake --install build --config Release

:: 2. Build the ROS2 package
call D:\Dev\ros2-windows\setup.bat
pixi run --manifest-path D:\Dev\ros2-windows\pixi.toml colcon build ^
  --packages-select ame_ros2 --base-paths ros2 ^
  --cmake-args "-DCMAKE_PREFIX_PATH=D:/Dev/ros2-windows;D:/Dev/repo/ame/build/install"
```

## Run

**Quick (use `run_ros2.bat`):**
```bat
call D:\Dev\repo\ame\run_ros2.bat
```

**In-process demo:**
```bat
call D:\Dev\ros2-windows\setup.bat
call D:\Dev\repo\ame\install\setup.bat
ros2 launch ame_ros2 ame_inprocess.launch.py ^
  pddl_file:=D:/Dev/repo/ame/domains/uav_search/domain.pddl ^
  problem_file:=D:/Dev/repo/ame/domains/uav_search/problem.pddl
```

**Send a plan goal:**
```bat
ros2 action send_goal /planner_node/plan ame_ros2/action/Plan ^
  "{goal_fluents: ['(searched sector_a)', '(classified sector_a)'], replan: false}"
```

**Load a domain at runtime (devenv/testing):**
```bat
ros2 service call /world_model_node/load_domain ame_ros2/srv/LoadDomain ^
  "{domain_id: 'uav_search', domain_pddl: '(define (domain ...))', problem_pddl: '(define (problem ...))'}"
ros2 service call /planner_node/load_domain ame_ros2/srv/LoadDomain ^
  "{domain_id: 'uav_search', domain_pddl: '(define (domain ...))', problem_pddl: '(define (problem ...))'}"
```

# A-GRA P3 proto → SysML XMI (reverse MBSE) plan

**Status:** proposed, not yet scheduled.
**Owner:** PYRAMID PIM tooling.
**Relates to:** the forward MBSE pipeline in
[`subprojects/PYRAMID/pim/mbse/`](../../../subprojects/PYRAMID/pim/mbse/)
and the A-GRA P3 seam in
[`subprojects/PYRAMID/pim/agra_p3_seam/`](../../../subprojects/PYRAMID/pim/agra_p3_seam/).

## 1. Purpose

Today the model-based systems engineering (MBSE) tooling runs in one
direction only: a SysML model authored in Cameo Systems Modeler is exported
as XMI and turned into Protocol Buffer (`.proto`) contract files. This plan
designs the opposite direction — take the A-GRA P3 `.proto` contract and
reconstruct a SysML model, exported as XMI, that a systems engineer can open
in Cameo. The goal is a proto that was written or generated outside the SysML
tool can be pulled back into the authoritative model so the two
representations stay in step.

"XMI" is the OMG XML Metadata Interchange format: the file format Cameo (and
other UML/SysML tools) use to save and exchange a model. "SysML" is the
Systems Modeling Language, a UML profile. The forward tooling consumes a
Cameo XMI export; the reverse tooling should produce a file Cameo can import.

## 2. What the forward pipeline does (so we can invert it)

The forward direction is two independent stages joined by a JSON document:

1. **`mbse/sysml_parser.py`** reads a Cameo XMI export and writes an
   intermediate JSON model. The JSON has five top-level lists —
   `dataTypes`, `enumerations`, `classes`, `connectors`, and `refinements`.
   Every element keeps its XMI identifier (`id`), display `name`, package
   path (`namespace`), abstract flag, `«Block»`/`«InterfaceBlock»`
   stereotype, properties (with type reference, multiplicity, flow
   direction, and documentation), ports, generalizations, and `«Refine»`
   edges.

2. **`mbse/proto_generator.py`** reads that JSON model and writes `.proto`
   files. The important transforms it performs (each of which the reverse
   tool must undo) are:

   | Forward transform | Where |
   |---|---|
   | DataTypes/classes → messages; enumerations → enums | `_write_message`, `_write_enum` |
   | Package `pyramid.data_model.<project>[.<component>]` derived from namespace | `_data_package` |
   | Generalization (inheritance) flattened to a composed `base` / `base_N` field | `_write_message` inheritance block |
   | A field typed by an abstract type expanded to a `oneof` over its concrete children | `_write_message` oneof block |
   | `«InterfaceBlock»` classes realised as `service`s, never as messages | `_write_message_from_class`, `_write_one_port_service` |
   | Component ports → services in `pyramid.components.<project>[.<component>].services.{provided,consumed}` | `_service_package`, `_write_service_file` |
   | Port kind (`request` / `information`) resolved by walking the generalization and `«Refine»` chain to `RequestService` / `ProviderService` | `sysml_parser._resolve_port_kind` |
   | Request port → `Create`/`Read`/`Update`/`Cancel` RPCs with `Ack`/`Query`/`Identifier`; information port → `Read(Empty)` streaming | `_write_one_port_service` |
   | Payload interface bound to a port → `_Request` / `_Entity` / `_Information` wrapper message with a `oneof` over concrete payload variants | `_payload_message`, `_classify_iface_payloads` |
   | Interaction pattern, topic, and QoS stamped as a `pyramid.options.pyramid_op` method option | `_write_interaction_option` |
   | Base value types (`Identifier`, `Timestamp`, `Angle`, …) auto-filled from a built-in table | `BASE_TYPE_DEFINITIONS` |
   | `CamelCase` property → `snake_case` field | `_to_snake_case` |

The reverse tool inverts this table. A useful front-end already exists:
**`pim/proto_parser.py`** parses a `.proto` tree into a structured model
(`ProtoFile` / `ProtoMessage` / `ProtoField` / `ProtoService` / `ProtoRpc`,
plus a cross-file `ProtoTypeIndex`) and already includes
`classify_port_service`, which recognises the `request` and `information`
port-service shapes. The reverse pipeline reuses it unchanged as stage 1.

## 3. A crucial nuance: the A-GRA P3 proto is not MBSE-generated

The P3 data-model proto header reads *"Generated by pim/xsd2proto.py"* and
the service protos read *"Generated by pim/gen_interaction_seam.py"*. In
other words the A-GRA P3 contract was produced from the A-GRA XSD and the
Table 3-1 interaction seam, **not** by `mbse/proto_generator.py`. Two
consequences follow, and they shape the whole plan:

- The reverse tool cannot assume the proto carries MBSE-shaped provenance.
  For example, the `base = 1` field in a P3 data message comes from an XSD
  `extension base`, not from a SysML generalization — although, conveniently,
  both should map back to a SysML generalization. The tool must recognise
  the *shape*, not trust a generator fingerprint.
- There is **no original SysML model and no source `.xmi` in the
  repository** to diff against. The A-GRA P3 model never existed in SysML.
  So "correctness" cannot mean "byte-identical to the input XMI"; it must be
  defined as semantic round-trip fidelity through the existing forward tools
  (Section 7).

This is why the deliverable is framed as *lifting* a proto contract into a
newly-synthesised SysML model, robust to protos the forward MBSE generator
never emitted, rather than a narrow bit-exact inverse of one generator.

## 4. Design: a symmetric two-stage reverse pipeline

Mirror the forward pipeline, sharing the same intermediate JSON model as the
pivot so the two directions meet in a documented, testable middle:

```
.proto tree ──▶ [proto_parser.py] ──▶ ProtoFile model
                                          │
                                          ▼
                            [proto_to_model.py]  (NEW, stage A)
                                          │
                                          ▼
                     intermediate JSON model  (same schema sysml_parser emits)
                                          │
                                          ▼
                            [xmi_generator.py]   (NEW, stage B)
                                          │
                                          ▼
                                   Cameo-importable .xmi
```

Two new modules, both under `subprojects/PYRAMID/pim/mbse/`:

- **Stage A — `proto_to_model.py`**: turn the parsed `ProtoFile` set into
  the intermediate JSON model. This is the semantic inverse of
  `proto_generator.py` and holds almost all of the difficulty.
- **Stage B — `xmi_generator.py`**: serialise the intermediate JSON model to
  SysML XMI. This is the mechanical inverse of `sysml_parser.py`; the
  difficulty here is XMI-flavour fidelity, not semantics.

Choosing the intermediate JSON as the pivot (rather than proto → XMI in one
shot) gives three things: the reverse tool is testable at the JSON boundary
against the same schema the forward tool already produces; stage B is reusable
for any future "model → XMI" need; and the lossy inversions are isolated in
stage A where they can be unit-tested in plain data.

## 5. Reverse mapping (stage A, proto → model)

| Proto construct | Reconstructed model element | Notes / difficulty |
|---|---|---|
| top-level `message` in `pyramid.data_model.*` | `dataTypes[]` entry (a `«Block»`/DataType) | package → `namespace` via the inverse of `_data_package` |
| `enum` | `enumerations[]` entry | strip the `SCREAMING_SNAKE` prefix and the synthetic `_UNSPECIFIED = 0` literal |
| scalar field (`double`, `string`, …) | property with a primitive type reference | inverse of `TYPE_MAPPING` |
| message-typed field | property referencing another type by id | resolve via `ProtoTypeIndex` |
| `repeated` / `optional` field | multiplicity `0..*` / `0..1` | inverse of the multiplicity rules |
| leading `base` / `base_N` field whose type is a message | a **generalization** (not a property) | see 6(a) — the central heuristic |
| `oneof` whose arms are concrete subtypes of one abstract type | an **abstract supertype property** + generalization edges from each arm | see 6(b) — lossy, supertype name is not in the proto |
| `service Foo_Service` | an `«InterfaceBlock»` + a component `«Block»` with a typed port | inverse of `_write_one_port_service` |
| `Create/Read/Update/Cancel` shape (Ack/Query/Identifier) | a `request` port refining `RequestService` | use `classify_port_service` → `'request'` |
| `Read(Empty) returns (stream …)` shape | an `information` port refining `ProviderService` | use `classify_port_service` → `'information'` |
| `…_Request` / `…_Entity` / `…_Information` wrapper `oneof` | the interface block's bound payload properties (request/entity/information roles) | inverse of `_payload_message` / `_classify_iface_payloads` |
| service package segment `…services.provided` / `…services.consumed` | port `direction` (and hence port `isConjugated`) | consumed ⇒ conjugated |
| `pyramid.options.pyramid_op` (pattern/topic/qos) | retained as documentation/tagged values on the port or its RPCs | recover topic and QoS for traceability |
| `google.protobuf.Timestamp` field | `Timestamp` base value type reference | inverse of the `use_google_timestamp` special case |
| `google.protobuf.Empty` payload | an **absent** payload (no property) | do not synthesise an `Empty` type |
| `Identifier` / `Ack` / `Query` support types (port grammar) | recognised base types, not re-modelled as domain data | keep them in a `base`/`port_grammar` package |
| `snake_case` field name | `CamelCase` property name | inverse of `_to_snake_case`; imperfect (see 6(c)) |

## 6. Lossy inversions and the decisions they force

Several forward transforms discard information. Each needs an explicit,
documented decision; these are the load-bearing parts of the plan and the
places reviewers should focus.

**(a) Inheritance was flattened to a composed `base` field.** The reverse
tool must recognise a leading field named `base` (or `base_2`, `base_3`, …)
whose type is a message and lift it back to a SysML generalization rather
than emitting it as an ordinary property. Decision: treat a first field
literally named `base`/`base_N` and typed by a message as a generalization
edge; emit a diagnostic when a message has a field named `base` that does
*not* fit the pattern, so a genuinely data-named `base` field is never
silently reinterpreted. This rule also correctly recovers the A-GRA XSD
`extension base` composition, which uses the same `base = 1` shape.

**(b) Abstract-type fields were expanded to a `oneof`; the abstract
supertype name was lost.** A forward `oneof` over concrete children carries
no record of the abstract parent it replaced. Decision: reconstruct a
synthetic abstract supertype for each such `oneof`, name it deterministically
(for example from the field/oneof name), mark it `isAbstract`, and attach a
generalization from each arm to it. Where a project convention or a sidecar
map (6(e)) supplies the real supertype name, prefer that. Document that the
abstract type name in the regenerated model is synthetic unless recovered.

**(c) `CamelCase` → `snake_case` is not perfectly invertible.** `snake_to_pascal`
already exists in `proto_parser.py`, but acronyms and digit boundaries are
ambiguous (`amti_command` → `AmtiCommand` vs `AMTI_Command`). Decision: use
the deterministic `snake_to_pascal` inverse as the default, and allow an
optional name-casing override table (seeded from the actual A-GRA type names,
which are already present in the message names) so domain acronyms round-trip.

**(d) Base value types were auto-filled from a built-in table.** `Identifier`,
`Timestamp`, `Angle`, `Length`, and the port-grammar `Ack`/`Query` were
synthesised by the forward generator, not authored. Decision: recognise these
by name/shape and place them in the shared `base` / `port_grammar` packages as
base types, rather than re-deriving them as ordinary project messages. Keep
the recognised set in one table so it stays aligned with
`BASE_TYPE_DEFINITIONS`.

**(e) Optional provenance sidecar for exact round-trip.** For protos that
*are* MBSE-generated in future, the forward generator could emit a small
sidecar map (XMI id ↔ fully-qualified proto type, original property casing,
abstract-supertype names). When present, stage A consumes it to reconstruct
the original ids and names exactly; when absent (the A-GRA P3 case) the tool
falls back to the heuristics above. This is proposed as an optional Phase 4
enhancement, not a prerequisite.

**(f) Stable identity.** Proto has no XMI ids. Decision: mint deterministic
ids by hashing the fully-qualified proto type name (and member name), so
regenerating the XMI from an unchanged proto yields an unchanged model and a
clean diff. A UUID-shaped derivation keeps Cameo happy.

## 7. Validation strategy

Because no original XMI exists to diff against, correctness is defined as
**semantic round-trip fidelity through the existing forward tools**, layered:

1. **Structural invariants (fast, in CI).** After stage A, assert counts and
   relationships hold: every data-model message maps to exactly one
   dataType/class; every classified service maps to one port with the
   expected kind and direction; every enum round-trips its literals.

2. **Full round-trip (the primary gate).**
   `proto  →  [reverse]  →  XMI  →  sysml_parser.py  →  JSON  →
   proto_generator.py  →  proto'`, then compare `proto` with `proto'`
   **semantically** (normalise formatting, field ordering, and the known
   lossy points from Section 6). The forward tools are trusted, so any
   surviving difference is a reverse-tool defect. Start the bar at the A-GRA
   P3 service protos (the port-grammar shapes, which are the highest-value and
   best-specified) and widen to the full 2856-message data model.

3. **Model-JSON round-trip (isolates stage B).**
   `JSON  →  xmi_generator.py  →  sysml_parser.py  →  JSON'` and diff the two
   JSON documents. This pins stage B (XMI emission) independently of the
   lossy stage A.

4. **Cameo import (acceptance, manual/out of CI).** Import the generated XMI
   into Cameo Systems Modeler and confirm packages, blocks, interface blocks,
   ports, and generalizations render. This is the real-world gate for XMI
   flavour and is recorded as evidence, not automated.

New tests sit beside the existing ones: `test_proto_to_model.py` (stage A
unit + structural), `test_xmi_generator.py` (stage B + JSON round-trip), and
`test_agra_p3_reverse.py` (the full P3 round-trip gate), following the
pattern in `test_proto_generator.py` and `test_proto_parser.py`.

## 8. Phased delivery

- **Phase 0 — round-trip harness and fixtures.** Stand up the semantic proto
  comparison and the model-JSON diff used by every later phase. Seed a tiny
  hand-written proto fixture whose expected model is known, so the pipeline is
  testable before it is complete.

- **Phase 1 — data model (messages/enums) round-trip.** Stage A for
  `pyramid.data_model.*`: messages, enums, fields, multiplicity, the `base`
  generalization lift (6a), base-type recognition (6d), and deterministic ids
  (6f). Stage B emits DataTypes, Enumerations, packages, generalizations, and
  documentation as XMI. Gate: the A-GRA P3 data model round-trips
  semantically.

- **Phase 2 — services and ports.** Stage A for `pyramid.components.*`:
  invert the port-service shapes using `classify_port_service`, recover the
  interface block and its request/entity/information payloads, port kind,
  direction, and the interaction option (topic/QoS). Stage B emits
  `«InterfaceBlock»`/`«Block»` stereotype applications, ports (with
  `isConjugated`), FlowProperties, connectors, and `«Refine»` edges. Gate:
  the A-GRA P3 service protos round-trip semantically.

- **Phase 3 — abstract/`oneof` reconstruction and full P3.** Implement the
  synthetic-supertype reconstruction (6b) and the name-casing override (6c),
  then run the whole A-GRA P3 seam through the round-trip gate and drive
  differences to zero (modulo the documented lossy points). Deliver a small
  CLI (`reverse_mbse.py`) that runs proto-tree → XMI end to end.

- **Phase 4 (optional) — provenance sidecar for exact round-trip.** Add the
  forward-generator sidecar (6e) so MBSE-generated protos reconstruct their
  original ids/names exactly, and add Cameo-import acceptance evidence.

## 10. Risks and open questions

- **XMI flavour fidelity.** Cameo is particular about namespaces, stereotype
  application, and the exact element shapes it will re-import. Phase 2/3 must
  be validated by a real Cameo import (Phase 4 acceptance). Mitigation: model
  stage B's output on a *known-good* Cameo export of a small hand-built model
  so the target shape is copied, not guessed. **Open question: is a sample
  Cameo XMI export available to use as the shape reference?** (None is in the
  repo today; the forward parser's expectations in `sysml_parser.py` are the
  only in-repo specification of the flavour.)

- **Scale.** The P3 data model is ~58k lines / 2856 messages from 343 roots.
  Stage A must stay near-linear and the XMI writer must stream, not hold a DOM
  for the whole model if memory is tight.

- **Ambiguous inverses.** Sections 6(a)–(c) are heuristic. The plan's stance
  is: make each heuristic explicit, emit a diagnostic wherever a shape is
  ambiguous, and accept a *semantically equivalent* (not byte-identical) model
  unless the optional sidecar (6e) is present.

- **Support-type placement.** Whether the port-grammar `Ack`/`Query`/
  `Identifier` and the base value types should live in the reconstructed model
  as authored base types or be treated as tool-owned library types is a
  modelling decision to confirm with the systems-engineering owner before
  Phase 2.

## 11. Out of scope

- Reversing the Ada/C++/ROS2 binding generators — this plan covers only the
  proto-to-SysML direction.
- Behavioural SysML (state machines, activities, sequence diagrams); the model
  is structural (blocks, interface blocks, ports, data types, enumerations).
- Live interop or MA L1 compliance claims; the P3 seam carries none today
  (see its README) and reversing it adds none.
